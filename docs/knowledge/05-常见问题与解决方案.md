# 常见问题与解决方案

## Q1：为什么 ability services 返回空数组？

**原因分析**：
1. 该能力下确实没有挂载任何服务
2. 用户账号尚未订购该能力，没有查看其服务菜单的权限
3. 该能力是「目录型」能力，实际服务挂在子分类下

**解决方案**：
- 先确认用户是否已订购该能力（`sdp ability my`）
- 如果未订购，引导用户到网页端完成订购
- 如果是目录型能力，尝试用 `sdp service search` 搜索相关 API

---

## Q2：为什么 service view 的 requestUrl 为空？

**原因分析**：
部分服务接口在目录列表中只展示了基础信息，详细的路由地址需要在服务详情页或文档中查看。

**解决方案**：
- 查看 `interfaceId` 字段，很多接口的调用路径格式为：
  ```
  /openapi/{serviceId}?method={interfaceId}
  ```
- 结合 `protocol` + `requestTypeText` 构造完整 URL
- 如果确实为空，建议用户联系平台管理员或服务负责人（`owner` / `contactNo`）

---

## Q3：如何知道某个能力是否适合我的业务？

**判断维度**：
1. **desc（描述）**：是否直接对应你的业务诉求？
2. **provider（提供方）**：是否是你可信任的内部部门或集团能力？
3. **catalogName（分类）**：是否落在你的目标技术领域？
4. **能力下的服务数量**：服务越多通常说明该能力越成熟

**建议流程**：
```
sdp ability search "关键词" → 筛选 3-5 个候选 → 
sdp ability view <id> 查看详情 → 
sdp ability services <id> 查看服务能力 → 
结合业务场景做最终决策
```

---

## Q4：为什么 app list 返回空？

**原因**：该用户账号还没有在山东能力开放平台创建过任何应用。

**解决方案**：
- 应用是能力的消费载体，用户需要先登录网页端创建应用
- 创建应用后，才能将能力授权给该应用使用
- AI 目前只能查询应用列表，创建应用需在浏览器中完成

---

## Q5：返回的 JSON 里有乱码怎么办？

**原因**：平台部分接口返回 GBK 编码的数据。

**解决方案**：
- CLI 工具已内置 GBK→UTF-8 自动转码逻辑
- 如果终端仍显示乱码，建议：
  - Windows 用户换用 Windows Terminal 或 PowerShell 7+
  - 检查终端编码设置为 UTF-8

---

## Q6：哪些操作可以在 CLI 完成，哪些必须去网页端？

| 操作 | CLI 支持 | 说明 |
|-----|---------|------|
| 查询能力/服务/应用 | ✅ | 完全支持 |
| 查看详情和示例 | ✅ | 完全支持 |
| 登录认证 | ✅ | 通过插件获取 token 后执行 |
| 订购能力 | ❌ | 需用户手动在网页端完成 |
| 创建应用 | ❌ | 需用户手动在网页端完成 |
| 授权能力给应用 | ❌ | 需用户手动在网页端完成（需 BOMC 工单） |

---

## Q7：如何快速找到某个领域的所有能力？

**策略**：
1. 先用 `sdp ability list --size 100` 查看分类分布
2. 根据 `category` 字段过滤，如 "机器视觉"、"短信触达"、"定位服务"
3. 如果分类不够细，再用 `sdp ability search "关键词"` 补充

---

## Q8：AI 推荐的方案和实际有出入怎么办？

**原因**：
- 平台能力目录会随业务调整而变化
- 部分能力有白名单、订购状态、版本差异等限制
- AI 基于公开描述做推荐，无法获取账号级别的订购权限信息

**建议**：
- AI 推荐后，用户应在网页端二次确认能力的可用性和订购条件
- 对于关键业务，建议联系能力的 `owner` 或 `department` 做技术对接
