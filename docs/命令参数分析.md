# lc 命令参数分析

本文档整理当前支持的所有命令及其必要参数，分析参数获取方式，提出简化建议。

---

## 详细命令列表

### 1. space（研发空间）

| 子命令 | 必要参数 | 自动探测 | 说明 |
|--------|----------|----------|------|
| `list` | 无 | ❌ | 查询所有研发空间 |
| `project available` | 无 | ❌ | 查询可关联的项目 |
| `project linked` | `--workspace-key` | ✅ | 查询已关联的项目 |

**参数获取：**
```bash
# workspace-key 获取
lc space list  # 返回 spaceCode 字段
```

|------|------|----------|----------------|
| workspace-key | `-w` | `lc space list` | ✅ |

| git-project-id | `--git-project-id` | `lc repo list` | ❌ |
| project-code | `-p` | `lc space project linked` | ❌ |

### 对象ID类参数（操作特定对象时需要）

| 对象 | ID名称 | 获取方式 |
|------|--------|----------|
| 需求 | object-id | `lc req list` |
| 任务 | task-id | `lc task list` |
| 缺陷 | defect-id | `lc bug list` |
| MR | mr-id (iid) | `lc pr list` |

## 参数简化分析

### 现状问题

1. **重复参数多**
   - 大多数命令都需要 `--workspace-key`
   - ~~查询类命令曾需 `--workspace-name`~~（已优化：workspace-name 可自动从 workspace-key 获取，已删除此参数）

2. **参数获取链长**
   ```
   示例：创建需求
   1. lc space list → 获取 workspace-key
   2. lc space project linked -w <key> → 获取 project-code
   3. lc req create "name" -w <key> -p <code>
   ```

3. **命名不一致**
   - `--project-code` vs `--project-id` vs `-p`
   - `--workspace-key` vs `-w`

### 已实现的优化

✅ **自动探测**（已实现）
- `req list` / `bug list` / `space project linked` 支持
- 在Git仓库目录下自动获取 workspace-key

✅ **环境变量**（已支持）
- `LC_WORKSPACE_KEY`
- `LC_WORKSPACE_NAME`

### 可进一步简化的方向

#### 1. 扩展自动探测支持

**当前支持自动探测的命令：**
- `req list`
- `bug list`
- `space project linked`

**建议扩展：**
| 命令 | 可行性 | 优先级 |
|------|--------|--------|
| `task list` | 高 | 高 |
| `repo list` | 高 | 中 |
| `pr list` | 高 | 中 |
| `req create` | 中 | 中 |
| `task create` | 中 | 中 |
| `bug create` | 低 | 低 |

**技术方案：**
```go
// 在 task list 中增加自动探测
taskListCmd.Run = func(cmd *cobra.Command, args []string) {
    // 尝试自动探测
    if taskListWorkspaceKey == "" {
        autoResult := common.TryAutoDetect(true)
        if autoResult.Success {
            taskListWorkspaceKey = autoResult.Context.WorkspaceKey
        }
    }
    listTasks()
}
```

#### 2. 配置文件缓存

**方案：本地项目配置 `.lc/config.json`**

```json
{
  "workspaceKey": "XXJSLJCLIDEV",
  "workspaceName": "灵畿cli研发",
  "projectCode": "XXJSLJCLIDEV",
  "gitProjectId": "44618"
}
```

**优先级：**
1. 显式参数（最高）
2. 环境变量
3. 本地配置文件
4. 自动探测
5. 交互式提示（最低）

**实现位置：** `internal/config/local_config.go`

#### 3. 交互式参数补全

**方案：缺少必需参数时交互式询问**

```bash
$ lc req create "新需求"
未指定 --workspace-key，请选择：
1. XXJSLJCLIDEV (灵畿cli研发)
2. XXJSxiaobaice (小白测研发项目)
> 1

未指定 --project-code，请选择：
1. XXJSLJCLIDEV
> 1

创建成功！
```

**技术方案：** 使用 `survey` 或 `promptui` 库

#### 4. 快捷命令/别名

**方案：常用workflow封装**

```bash
# 当前方式（3步）
lc space list
lc space project linked -w XXJSxiaobaice
lc req create "需求" -w XXJSxiaobaice -p R24113J3C04

# 优化方式（1步）
lc quick req-create "需求"  # 自动使用当前目录的workspace和第一个project
```

#### 5. 参数别名统一

**建议标准化：**

| 参数 | 长形式 | 短形式 | 现状 |
|------|--------|--------|------|
| 研发空间Key | `--workspace-key` | `-w` | ✅ 统一 |
| 项目代码 | `--project-code` | | ⚠️ 部分命令使用 `--project-id` |
| Git项目ID | `--git-project-id` | 无 | ⚠️ 无短形式 |
| 格式化输出 | `--pretty` | 无 | ✅ 统一 |

## 推荐简化方案（优先级排序）

### 高优先级（立即实施）

1. **扩展自动探测到 task list**
   - 影响大，改动小
   - 与 req list / bug list 保持一致

2. **统一参数命名**
   - bug命令的 `--project-id` 改为 `--project-code`
   - 保持与 req/task 一致

### 中优先级（近期实施）

3. **本地项目配置文件**
   - 创建 `.lc/config.json` 支持
   - 优先级：显式 > 环境变量 > 本地配置 > 自动探测

4. **为 pr 命令添加自动探测**
   - 如果在Git仓库，自动填充 git-project-id
   - 需要额外存储 gitProjectId 到本地配置

### 低优先级（未来考虑）

5. **交互式参数补全**
   - 开发成本高
   - 主要用于新用户

6. **快捷命令**
   - 使用场景有限
   - 可能增加复杂度
