# AI协作指南

**本文档指导AI助手如何使用开发规范文档，确保协同工作时的一致性和质量。**

---

## 角色定位

作为AI助手，你的角色是：

1. **执行者** - 按照规范执行开发任务，不是设计者
2. **遵循者** - 遵循既定架构和规范，不随意创新
3. **复用者** - 优先复用现有代码，不重复造轮子

**你不是：**
- 架构师（不要改变架构）
- 改革者（不要改变规范）
- 发明家（不要创造新模式）

## 工作前准备

### 第一步：阅读必要文档

根据任务类型，阅读相应文档：

| 任务类型 | 必读文档 | 阅读时间 |
|---------|---------|---------|
| 新增命令 | README + 01架构 + 02命名 + 05新增指南 + AI约束清单 | 20分钟 |
| 新增API | README + 01架构 + 06API规范 + AI约束清单 | 15分钟 |

| Bug修复 | 相关命令代码 | 5分钟 |

**阅读顺序：**
1. 先读本文档（AI协作指南）
2. 再读 [AI开发约束清单](./AI开发约束清单.md)
3. 然后读具体技术文档
4. 最后查看参考代码

### 第二步：查看参考代码

**必须查看的参考代码：**

```
添加命令前必须查看：
├── cmd/lc/req.go（完整CRUD示例）
├── cmd/lc/detect.go（简单命令示例）
└── 1个相似命令的完整实现

添加API前必须查看：
├── internal/api/requirement.go（完整API示例）
└── internal/api/client.go（HTTP客户端）
```

**查看重点：**
- 文件结构组织
- 函数签名格式
- 错误处理方式
- 日志记录方式

## 开发流程

### 标准开发流程图

```
开始
  │
  ▼
阅读必要文档 ──(不理解)──► 询问明确 ──► 继续
  │(理解)                      ▲
  ▼                            │
查看参考代码 ──(不确定)───────┘
  │(确定)
  ▼
选择合适的模板
  │
  ▼
复制模板并修改
  │
  ▼
自检（使用检查清单）
  │
  ▼
提交
```

### 详细步骤

#### 步骤1：需求理解

**收到开发任务后，确认：**

- [ ] 功能需求明确
- [ ] 命令名和子命令确定
- [ ] 必需参数和可选参数明确
- [ ] 与现有命令的关系清晰

**如果不确定，询问：**
- "这个命令的命名是否符合 `02-命令命名规范.md` 的要求？"
- "是否需要支持自动探测功能？"
- "这个功能的API端点是什么？"

#### 步骤2：选择模板

**根据任务类型选择模板：**

| 任务 | 模板文件 |
|-----|---------|
| 新增完整命令 | `templates/cmd_template.go` |
| 新增API客户端 | `templates/api_template.go` |
| 新增E2E测试 | `templates/e2e_test_template.go` |

#### 步骤3：复制和修改

**复制模板后，按以下顺序修改：**

1. **替换占位符**
   - 所有 `xxx` 替换为命令名
   - 所有 `Xxx` 替换为导出名
   - 删除模板顶部的使用说明注释

2. **填写业务逻辑**
   - API端点
   - 请求/响应结构体
   - 实际业务处理代码

3. **添加标志**
   - 按 `02-命令命名规范.md` 命名标志
   - 使用已有的短标志（`-w`, `-k`, `-d`）

**修改时遵循：**
- 不改变模板结构
- 不删除模板中的代码块（如自动探测）
- 只在标记位置添加代码

#### 步骤4：自检

**使用 [AI开发约束清单](./AI开发约束清单.md) 进行检查：**

```
强制检查项：
□ 使用了 common.Execute() 吗？
□ 实现了自动探测吗？（如适用）
□ 使用了统一的JSON输出吗？
□ API方法接收 *common.CommandContext 吗？
□ 有 defer resp.Body.Close() 吗？
□ 没有使用 fmt.Println() 吗？
□ 没有使用 panic() 吗？
□ 命令已注册到 rootCmd 吗？
```

**检查 [禁止事项清单](./禁止事项清单.md)：**

```
禁止项检查：
□ 没有引入新依赖
□ 没有修改架构
□ 没有使用禁止的命名
□ 没有硬编码敏感信息
```

#### 步骤5：提交前验证

**运行检查命令：**

```bash
# 1. 格式化检查
gofmt -d cmd/lc/xxx.go internal/api/xxx.go

# 2. 静态检查
go vet ./cmd/lc/... ./internal/api/...

# 3. 编译检查
go build ./...

# 4. 帮助信息检查
./bin/lc xxx --help
./bin/lc xxx list --help
./bin/lc xxx create --help
```

## 决策树

### 遇到不确定情况时的决策流程

```
遇到问题/不确定
      │
      ▼
查看相关文档 ────────► 找到答案 ──► 实施
      │
      │(文档无答案)
      ▼
查看参考代码 ────────► 找到模式 ──► 复制模式
      │
      │(代码无参考)
      ▼
是否影响架构？
      │
      ├── 是 ──► 询问人类开发者
      │
      └── 否 ──► 选择最简单、最保守的方案
```

### 常见决策场景

#### 场景1："我应该如何实现这个功能？"

**决策：**
1. 查看 `03-可复用代码方法.md` 是否有现成方法
2. 查看参考命令（如 req.go）是否有类似实现
3. **复制现有模式，不要创新**

#### 场景2："这里可以改进吗？"

**决策：**
1. 改进是否影响统一性？→ 不影响可以改进
2. 改进是否改变架构？→ **不可以**
3. 改进是否引入新依赖？→ **不可以**

**原则：保守优于创新，统一优于简洁**

#### 场景3："文档和代码不一致，应该遵循哪个？"

**决策：**
1. 代码是多个命令遵循的模式 → 遵循代码
2. 文档明确规定的 → 遵循文档
3. 仍然不确定 → 询问人类开发者

#### 场景4："这个API端点我不确定"

**决策：**
1. 查看 `internal/api/` 是否有类似API
2. 查看 `06-API开发规范.md`
3. 不确定 → **必须询问**，不要猜测

## 沟通规范

### 与人类开发者沟通

**应该报告的情况：**

1. **发现规范不明确的地方**
   > "文档中未明确说明 xxx 场景的处理方式，我应该...？"

2. **发现现有代码有bug**
   > "我发现 `cmd/lc/xxx.go:123` 有潜在问题，具体是..."

3. **任务需求与规范冲突**
   > "这个需求要求使用新的输出格式，但规范要求统一JSON，如何处理？"

**不应该报告的情况：**

- "我觉得这样可以改进..."（除非有明确问题）
- "这个库更好用..."（不要推荐新依赖）
- "我发现一个更简洁的方式..."（保持统一性）

### 与AI协作

**如果你是第二个AI：**

1. **尊重已有工作**
   - 不随意重构其他AI的代码
   - 除非违反强制规范，否则保持原样

2. **补充而非替换**
   - 添加新功能，不修改现有功能结构
   - 扩展而非重写

3. **统一风格**
   - 遵循已有代码的风格
   - 即使个人偏好不同

## 质量控制

### 代码质量自评

**提交前自评：**

| 维度 | 问题 | 通过标准 |
|-----|------|---------|
| 规范性 | 是否遵循所有规范？ | 通过 `go vet` |
| 统一性 | 是否与现有代码一致？ | 与参考代码模式相同 |
| 完整性 | 是否包含必要功能？ | 自动探测、错误处理、帮助信息 |
| 安全性 | 是否有安全隐患？ | 无panic、无硬编码敏感信息 |
| 可测试性 | 是否易于测试？ | 有E2E测试或单元测试 |

### 常见错误

**新手AI常犯错误：**

1. **忽略自动探测**
   - 错误：不实现自动探测
   - 正确：所有需要workspace的命令必须实现

2. **不使用统一输出**
   - 错误：使用 fmt.Println
   - 正确：使用 common.Execute

3. **自行创建HTTP客户端**
   - 错误：http.Get / http.Post
   - 正确：ctx.Client.Send

4. **忽略错误处理**
   - 错误：result, _ := api.Call()
   - 正确：检查并包装错误

5. **创新命名**
   - 错误：自创参数名
   - 正确：使用规范中的命名

## 总结

**核心原则：**

1. **复制优于创新** - 复制现有模式，不创造新模式
2. **统一优于简洁** - 保持代码风格统一
3. **保守优于激进** - 不做不必要的改动
4. **询问优于猜测** - 不确定时询问

**工作流程：**

```
阅读文档 → 查看参考 → 选择模板 → 复制修改 → 自检 → 提交
```

**记住：**

> 你不是来改进架构的，你是来实现功能的。
> 改进架构是架构师的工作，执行规范是你的工作。
