# 能力与服务订购流程指南

> 本文档面向 AI 助手，明确山东能力开放平台的两种核心订购路径、标准操作流程及常见约束，确保 AI 在协助用户完成订购时逻辑准确、不出错。

---

## 一、核心概念区分

在平台上，**能力（Ability）**和**服务（Service/API）**是两个不同层级的资源，订购逻辑也不同。

| 维度 | 能力订购 | 服务订购 |
|------|---------|---------|
| 资源层级 | 能力 = 一组相关 API 的集合 | 服务 = 单个具体接口/API |
| 入口 | 能力详情页 → 立即订购 | 数字服务详情页 → 立即申请 |
| 适用场景 | 对外项目，需要完整的能力组合 | 只需要调用某一个具体接口 |
| 订购后动作 | 需要在「我的应用」中做**能力授权** | 订购时直接选择应用，**一步完成** |
| 结果 | 该能力下**所有服务**对应用可见 | **仅该服务**对应用可见 |

### 关键记忆点
- **能力**：以 `CA` 开头（如 `CA202303051941169651007662311817`）
- **服务**：以 `SE` 开头（如 `SE202604100446026231000121`）
- 能力的订购和授权是**两次请求**；服务的订购和授权是**一次请求**。

---

## 二、路径 A：能力订购 + 能力授权

### 步骤 1：订购能力
执行命令：
```bash
sdp ability order <ability-id>
```

平台会记录一条订购申请，审批通过后该能力对用户账户可见。

### 步骤 2：将能力授权给应用
执行命令：
```bash
sdp app auth-ability <app-id> \
  --ability <ability-id> \
  --bomc <bomc-ticket> \
  --quota-limit 500 \
  --limit-count 500 \
  --policy-period 1 \
  --policy-time-unit second
```

**必填参数说明：**
- `app-id`：要授权的目标应用 ID
- `--ability`：能力 ID（以 CA 开头）
- `--bomc`：BOMC 工单号，平台审批所需

**流控参数说明：**
- `--quota-limit`：配额上限（默认 500）
- `--limit-count`：单位时间请求次数上限（默认 500）
- `--policy-period`：流控周期数值（默认 1）
- `--policy-time-unit`：流控周期单位（默认 `second`，可选 `minute`/`hour`/`day`）

> AI 必须根据用户提供的 BOMC 工单号执行授权，**禁止臆造**工单号。如果用户没有提供，必须主动询问。

---

## 三、路径 B：服务订购（一步完成授权）

执行命令：
```bash
sdp service order <service-id> \
  --app-id <app-id> \
  --bomc <bomc-ticket> \
  --quota-limit 100 \
  --limit-count 100 \
  --policy-period 1 \
  --policy-time-unit second
```

**特点：**
- 提交后同时完成「订购」和「应用授权」
- 无需再执行 `app auth-service` 之类的二次操作
- 流控参数默认值通常为 100/100/1/second

---

## 四、路径 C：查询申请/授权进展

执行命令：
```bash
sdp order list --size 20 --pass-status true
```

返回字段说明：
- `appName`：申请关联的应用名称
- `goodsName`：申请的能力/服务名称
- `authType`：`capacity` 表示能力，`api` 表示服务
- `status` / `statusName`：审批状态（如待审批、已通过、已拒绝）
- `applyTime`：申请时间

> 注意：`passStatus` 的语义由平台定义，`true` 通常表示查询已通过/已处理的记录，`false` 可能表示待审批。实际使用时可根据页面表现调整。

---

## 五、AI 操作决策树（必须遵守）

当用户说"我要订购 XXX"时，AI 必须按以下顺序判断：

```
1. 判断 XXX 是能力还是服务：
   - ID 以 CA 开头 或 用户明确说"能力" → 走路径 A
   - ID 以 SE 开头 或 用户明确说"服务/接口/API" → 走路径 B

2. 路径 A（能力）：
   a. 执行 sdp ability order <ability-id>
   b. 询问用户目标 app-id 和 BOMC 工单号
   c. 执行 sdp app auth-ability <app-id> --ability <ability-id> --bomc <工单号>

3. 路径 B（服务）：
   a. 询问用户目标 app-id 和 BOMC 工单号
   b. 执行 sdp service order <service-id> --app-id <app-id> --bomc <工单号>

4. 缺少必填参数时，AI 必须向用户询问，禁止臆造。
5. 每一步执行后汇报结果，确认成功后再执行下一步。
```

## 六、常见错误与规避

| 错误场景 | 正确做法 |
|---------|---------|
| 把能力 ID 当成服务去订购 | 先确认 ID 前缀：`CA` = 能力，`SE` = 服务 |
| 能力订购后忘记做应用授权 | 能力订购成功后必须执行 `app auth-ability` |
| 服务订购后再次去找「授权服务」入口 | 服务订购已一步完成，无需二次授权 |
| 未提供 BOMC 工单号就提交 | 必须询问用户，获取真实工单号 |
| 流控参数随意填写 | 无特殊需求时使用默认值即可 |

## 七、典型对话示例

### 示例 1：用户想订购能力
**用户**：我要订购地图坐标转换能力，应用是新员工实战应用，BOMC 工单号是 WO20260415001。

**AI 动作**：
```bash
sdp ability search "地图坐标转换"
# 确认 ability-id = CA202303051941169651007662311817
# 确认 app-id = 202508051556368141000509

sdp ability order CA202303051941169651007662311817
sdp app auth-ability 202508051556368141000509 \
  --ability CA202303051941169651007662311817 \
  --bomc WO20260415001
```

### 示例 2：用户想订购服务
**用户**：我要把营业厅信息查询接口授权给新员工实战应用。

**AI 动作**：
```bash
sdp service search "营业厅信息查询"
# 确认 service-id = SE202604100446026231000121
# 确认 app-id = 202508051556368141000509

sdp service order SE202604100446026231000121 \
  --app-id 202508051556368141000509 \
  --bomc <询问用户>
```
