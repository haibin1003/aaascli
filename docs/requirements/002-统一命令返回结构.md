# 1. 背景（Why）

当前LC命令的返回结构存在以下问题：

1. **返回格式不统一**：有的命令返回纯JSON对象，有的返回数组，有的包含`success`字段，有的不包含
2. **错误处理不一致**：有的命令在错误时返回JSON，有的直接打印错误信息并退出
3. **缺少元信息**：返回结果中缺少请求ID、执行时间、分页信息等元数据
4. **前端/脚本解析困难**：不统一的结构导致调用方难以统一处理响应

例如：
- `lc space list` 返回数组
- `lc req list` 返回 `{count, items, totalCount}`
- `lc req create` 返回 `{success, name, objectId, key, createdAt}`
- `lc repo list` 返回 `{pagination, repositories}`

因此，需要设计并实现统一的命令返回结构，提升工具的一致性和可维护性。

# 3. 非目标（Explicitly Out of Scope）

1. ❌ **不修改API层**：仅修改CLI命令层的输出格式，不涉及与灵畿平台的API交互
2. ❌ **不提供API版本**：本需求仅针对CLI工具，不提供HTTP API服务
3. ❌ **不支持自定义模板**：输出格式由工具定义，不支持用户自定义模板
4. ❌ **不向后兼容**：移除旧格式支持，所有输出统一为JSON格式
5. ❌ **不支持其他输出格式**：仅支持JSON，不支持table/yaml等其他格式

# 4. 使用场景 / 用户路径

## 场景1：脚本调用获取结构化数据

```bash
# 脚本中统一处理响应
result=$(lc req list -w XXJSLJCLIDEV -n "灵畿cli研发" --output json)
if [ $(echo $result | jq -r '.success') == "true" ]; then
    count=$(echo $result | jq -r '.data.count')
    echo "找到 $count 条需求"
fi
```

# 5. 功能需求清单（Checklist）

## 5.1 统一返回结构设计

- [ ] **基础结构定义**：所有命令返回必须包含以下字段
  ```json
  {
    "success": true|false,
    "data": {},
    "error": null|{},
    "meta": {}
  }
  ```

- [ ] **success字段**：布尔值，表示操作是否成功
- [ ] **data字段**：操作成功时的返回数据（对象或数组）
- [ ] **error字段**：操作失败时的错误信息（包含code、message、details）
- [ ] **meta字段**：元信息（包含requestId、timestamp、executionTime、version）

## 5.2 分页数据结构

- [ ] **统一pagination结构**：
  ```json
  {
    "pagination": {
      "page": 1,
      "pageSize": 20,
      "total": 156,
      "totalPages": 8,
      "hasNext": true,
      "hasPrev": false
    },
    "items": []
  }
  ```

## 5.3 输出格式

- [ ] **仅支持JSON格式**：所有命令输出统一为JSON格式
- [ ] **移除 `--output` 参数**：不再支持多格式切换

## 5.4 全局参数移除

- [ ] **移除 `--output` / `-o` 参数**
- [ ] **移除 `--no-header` 参数**
- [ ] **移除 `--fields` 参数**

# 6. 约束条件（非常关键）

## 6.1 技术约束

1. **不向后兼容**：移除所有旧格式输出，统一为JSON格式
2. **仅JSON输出**：所有命令必须输出标准JSON格式
3. **错误码规范**：错误码使用大写下划线格式，如`REQ_NOT_FOUND`、`INVALID_PARAMS`

## 6.2 架构约束

1. **统一封装层**：在`internal/common/output.go`中实现统一输出封装
2. **命令层改造**：各命令通过调用封装层输出结果，不直接操作stdout
3. **保留原始数据**：API返回的原始数据保留在`data.raw`字段中（可选）

## 6.3 字段命名规范

| 字段 | 命名规范 | 示例 |
|------|----------|------|
| 普通字段 | camelCase | `objectId`, `createdAt` |
| 常量/枚举 | 大写下划线 | `STATUS_NEW` |
| 错误码 | 大写下划线 | `REQ_NOT_FOUND` |
| 时间格式 | ISO 8601 | `2026-03-14T10:30:00Z` |

# 9. 验收标准（Acceptance Criteria）

### AC1：统一结构验证
**如果**执行任意LC命令并添加 `--output json`
**那么**返回的JSON必须包含`success`、`data`、`error`、`meta`四个顶级字段

### AC2：成功响应验证
**如果**命令执行成功
**那么**`success=true`，`data`包含业务数据，`error=null`

### AC3：失败响应验证
**如果**命令执行失败（如参数错误、资源不存在）
**那么**`success=false`，`data=null`，`error`包含错误码和错误信息

### AC4：表格格式验证
**如果**执行 `lc req list --output table`
**那么**输出格式化为表格，包含表头和数据行，便于阅读

### AC5：字段筛选验证
**如果**执行 `lc req list --fields=key,name,status`
**那么**仅显示指定的三个字段

### AC6：统一格式验证
**如果**执行任意LC命令
**那么**输出必须是标准JSON格式，非JSON格式视为缺陷

### AC7：所有命令改造完成
**如果**遍历所有LC命令
**那么**每个命令都支持新的统一返回结构

# 10. 风险与已知不确定点

| 风险点 | 影响 | 应对策略 |
|--------|------|----------|
| 存量脚本不兼容 | 高 | 通过版本升级说明告知用户，提供迁移指南 |
| 性能影响 | 低 | 格式化逻辑简单，性能损耗可忽略 |
| 错误码定义冲突 | 中 | 统一错误码前缀，如`REQ_`、`TASK_`、`REPO_` |

**不确定点**：
1. 是否需要支持输出到文件（`--output-file`）？
2. 表格格式是否需要支持颜色高亮？
3. 是否需要支持JSON Path过滤（如`-o json --jq '.data.items[0].name'`）？

# 11. 非目标

1. **不提供HTTP API**：本需求仅针对CLI工具输出格式
2. **不提供日志系统改造**：日志输出保持现有格式，不受统一结构影响
3. **不提供多语言支持**：输出消息保持中文，不考虑国际化
4. **不提供实时流式输出**：命令完成后一次性输出结果，不实时推送
5. **不提供图形界面**：仅CLI命令行界面
6. **不提供其他输出格式**：仅支持JSON，不支持table/yaml/csv等格式
7. **不向后兼容**：本次升级为破坏性变更，旧格式不再支持
