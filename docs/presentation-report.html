01 / 项目背景

为什么我们要做这款工具？

325+平台业务能力
3800+数字服务/API
数小时传统人工匹配耗时
30秒AI 辅助查询耗时

平台操作完全依赖浏览器

山东能力开放平台所有查询、订购、授权操作都必须在网页端完成，AI 助手无法直接调用平台接口，只能「口头指导」用户手动点击，导致 AI 的建议无法快速验证和落地。

人工查询效率极其低下

平台汇聚了 325+ 业务能力和 3800+ 具体 API 服务，业务人员需要在众多分类中逐层翻阅，纯人工浏览查找一个合适的能力往往耗费数小时，且容易遗漏更优选项。

海量能力与需求难以匹配

面对 325+ 业务能力和 3800+ API，业务人员不清楚如何根据自身需求筛选合适的能力，不知道哪些能力可以组合使用，更不清楚「能力」与「服务」的区别以及各自的适用场景。

拿到接口不会调用

即使找到了合适的 API 服务，业务人员也需要阅读冗长的接口文档，手动理解请求参数、响应格式和加密方式，拼凑调用代码的门槛高、耗时久、出错率高。

能力与服务的概念混淆

很多人对「能力（Ability）」和「服务（Service/API）」的层级关系、订购流程、授权方式一知半解，不清楚能力是对外项目使用还是内部系统调用，导致订购错误、授权遗漏、资源浪费。

AI 价值难以真正落地

AI 拥有强大的信息整合、逻辑推理和代码生成能力，但缺乏直接访问企业内部平台的通道，导致「有脑子、没手脚」；同时缺少规范约束，容易胡乱推荐、混淆概念，最终沦为一个不可靠的聊天工具。

💡

核心解决思路

为 AI 助手打造一款专用的命令行客户端（CLI），通过深入研究平台前端加密与通信机制，完整还原 RSA+AES 混合加密协议，让 AI 能够直接调用平台真实接口。同时，配套内置知识库、AI 使用手册、能力订购流程规范和 Prompt 模板，为 AI 提供「工具 + 知识 + 约束」三位一体的操作系统。

三大技术突破

🛡️ RSA+AES 混合加密还原

平台前端采用 RSA-2048 + AES-128-CBC-PKCS7 混合加密。我们通过逆向前端加密逻辑，在 Go 中完整实现了密钥生成、请求加密、响应解密的全流程，确保 CLI 与浏览器请求行为完全一致。

🔤 GBK 编码自动兼容

部分平台接口返回 GBK 编码的 JSON 数据，直接解析会出现中文乱码。我们内置了 GBK 到 UTF-8 的自动检测与转码机制，并将 golang.org/x/text 完整 Vendor 到本地，确保离线环境可用。

🍪 多 Cookie 联合认证

平台认证不仅依赖 Token，还需要 openPortalVerificationCode 和 openPortalServiceID 两个辅助 Cookie。我们通过 Chrome 扩展完整提取三 Cookie，并在 CLI 中自动注入到每个请求的 Header 和 Cookie 中。

不仅给 AI 递锤子，还教它怎么盖房子、给它画好施工图

🔧 工具层（CLI）

  • 直接查询 325+ 能力、3800+ API
  • 自动完成加密、认证、转码
  • 支持能力订购、服务订购、授权
  • 生成可直接运行的调用代码
🧠
AI
助手

📚 知识层（Knowledge）

  • 6 篇内置知识库文档（go:embed）
  • 典型业务场景和组合方案
  • 能力 vs 服务概念辨析
  • Python/Java/Go/cURL 代码模板

🎯 约束层（Skills & Prompt）

  • AI 使用手册：明确命令边界和行为规范
  • Prompt 模板：固定场景下的决策树约束
  • 订购流程规范：BOMC 工单、授权路径标准化
  • 防止臆造参数、混淆概念、胡乱推荐

为什么「三位一体」是核心亮点？

只有工具，没有知识，AI 每次都要从零推理，回答质量不稳定，新人上手慢；只有知识，没有工具，AI 只能纸上谈兵，无法验证和落地；工具+知识，没有约束，AI 容易「幻觉」——臆造参数、混淆能力与服务概念、跳过关键授权步骤。

我们的方案将 CLI 工具、内置知识库、规范约束融为一体：AI 先用知识库理解业务场景和能力差异，再用 CLI 直接查询和验证，最后被 Prompt 和流程规范约束在正确的决策路径内。三者相互补强，形成一个可执行、可验证、不易出错的AI 操作系统。

🛡️

关键优势：对原有系统零侵入

本项目无需修改平台后端一行代码、无需新增接口、无需调整数据库或网络策略。所有能力均通过纯客户端方式（命令行 + 浏览器插件）实现，对现有山东能力开放平台完全透明，风险可控，即插即用。
